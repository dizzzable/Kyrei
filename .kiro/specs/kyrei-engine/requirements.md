# Requirements Document

## Introduction

Kyrei — локальный (local-first) кроссплатформенный десктоп-агент для работы с кодом (Electron + Node/TypeScript). Цель этой спецификации — довести **движок** (`core/`) до состояния «лучше аналогов»: надёжнее правки, честнее контекст-менеджмент, безопаснее исполнение, гибче провайдеры, полностью офлайн-способный. Основано на исследовании `docs/research.md` §20–§28 (сравнение kilocode/opencode/codex/zed/kimi/MiMo/Mini-Agent/Antigravity/Kiro/pi-omp/Claude Code + AI SDK v5).

Принципы (незыблемые):
- **Local-first / offline-capable:** всё исполняется и хранится локально; дефолтный inference — Ollama (`localhost:11434/v1`). Внешние API — опционально.
- **Один писатель, read-рой для чтения** (Cognition-правило).
- **Restrictive-by-default безопасность:** никакого YOLO по умолчанию; workspace-jail на все инструменты.
- **Прозрачность и обратимость:** ничего не теряем безвозвратно (checkpoints/handoff), любые правки обратимы.
- **Фундамент:** AI SDK v5 (TS) тонкой обёрткой; Rust — только для CPU-hot подсистем позже (§29).

## Glossary

- **ход (turn)** — один запрос пользователя до финального ответа.
- **шаг (step)** — одна генерация модели внутри хода.
- **инструмент (tool)** — функция, вызываемая моделью.
- **провайдер** — источник модели (локальный/внешний).
- **jail** — ограничение файловых/командных операций пределами рабочей папки (с резолвом симлинков).
- **CCR** — Compress-Cache-Retrieve: обратимое сжатие контекста с восстановлением по идентификатору.

## Requirements

### Requirement 1: Фундамент движка на AI SDK v5 (локально)

**User Story:** Как разработчик Kyrei, я хочу, чтобы ядро агентного цикла работало на AI SDK v5 полностью локально, чтобы убрать хрупкий ручной SSE-код и не зависеть ни от какой чужой инфраструктуры.

#### Acceptance Criteria
1. WHEN движок инициализируется THE SYSTEM SHALL использовать `streamText` из пакета `ai` с явно переданным провайдер-объектом (никогда — модель голой строкой, чтобы исключить обращение к Vercel AI Gateway).
2. WHEN сконфигурирован локальный провайдер (Ollama/LM Studio/llama.cpp/vLLM) THE SYSTEM SHALL работать без интернета через `@ai-sdk/openai-compatible` с настраиваемым `baseURL`.
3. THE SYSTEM SHALL НЕ включать телеметрию (`experimental_telemetry` остаётся выключенной) и НЕ выполнять никаких сетевых запросов, кроме как к сконфигурированному `baseURL` провайдера.
4. WHEN движок собирается для дистрибуции THE SYSTEM SHALL включать зависимости `ai`, `@ai-sdk/openai-compatible`, `zod` в пакет так, чтобы приложение запускалось на Windows/macOS/Linux без внешних установок.
5. THE SYSTEM SHALL сохранять публичный контракт `runKyreiChat({ emit, messages, ... }) -> { text, parts }`, чтобы `gateway.js`, `session-store.js` и renderer не требовали изменений.
6. WHERE движок v2 включён через флаг `KYREI_ENGINE=v2` THE SYSTEM SHALL позволять параллельно оставлять рабочий движок v1 до полной верификации.

### Requirement 2: Протокол событий в стиле ACP

**User Story:** Как автор renderer и потенциальных интеграций (Zed/JetBrains), я хочу расширенный, стандартизуемый протокол событий, чтобы UI точно отражал жизненный цикл хода и отмена не выглядела как ошибка.

#### Acceptance Criteria
1. WHEN начинается ход THE SYSTEM SHALL эмитить `message.start`, затем серию `message.delta{text}` и `reasoning.delta{text}` по мере генерации.
2. WHEN модель вызывает инструмент THE SYSTEM SHALL эмитить жизненный цикл `tool.start` → (опц. прогресс) → `tool.complete` со **стабильным `tool_call_id`** от начала до конца.
3. WHEN ход завершается THE SYSTEM SHALL эмитить `message.complete{text,status,usage}` с явной причиной остановки `status ∈ {complete, interrupted, error, max_steps}` (аналог ACP `stopReason`).
4. WHEN пользователь отменяет ход THE SYSTEM SHALL завершать его как `status:"interrupted"` и НЕ эмитить событие `error` (отмена ≠ ошибка).
5. WHEN доступны данные использования THE SYSTEM SHALL сообщать `usage` (input/output токены; стоимость, если известна) в `message.complete` и промежуточно через `status.update`.
6. WHEN происходит нефатальная ошибка стрима THE SYSTEM SHALL эмитить `error{message}` в человеко-читаемом виде и, где возможно, продолжать.

### Requirement 3: Надёжный движок применения правок

**User Story:** Как пользователь, я хочу, чтобы правки кода применялись надёжно даже на «живых» файлах, чтобы агент не ломал файлы из-за сдвигов строк или пробелов.

#### Acceptance Criteria
1. THE SYSTEM SHALL поддерживать применение правок по **контекстным якорям** (окружающие строки), а не по номерам строк.
2. WHEN точное совпадение якоря не найдено THE SYSTEM SHALL применять толерантный поиск по убыванию строгости: (1) точное → (2) игнор trailing whitespace → (3) игнор leading+trailing whitespace → (4) нормализация Unicode-пунктуации.
3. IF якорь совпадает более чем в одном месте THE SYSTEM SHALL отклонять правку с actionable-ошибкой (требовать больше контекста), а НЕ применять молча к нескольким местам.
4. WHEN правка применяется THE SYSTEM SHALL вычислять построчный дифф (для UI `inline_diff`) и включать счётчик `+N −M`.
5. WHERE файл меньше ~400 строк THE SYSTEM SHALL позволять полную перезапись (`write_file`); WHERE больше — предпочитать точечную правку (`edit_file`).
6. WHEN модель прислала «грязный» ввод (heredoc-обёртки, лишние пробелы) THE SYSTEM SHALL лениво (lenient) распознавать намерение вместо жёсткого отказа.
7. WHEN перед правкой THE SYSTEM SHALL создавать обратимый снапшот (git-snapshot или копия), чтобы правка откатывалась.
8. THE SYSTEM SHALL сохранять исходный стиль перевода строк файла (LF/CRLF/mixed) и наличие/отсутствие финального перевода строки при записи; нормализация применяется ТОЛЬКО для сопоставления якоря, но НЕ для записываемых байтов.
9. THE SYSTEM SHALL сохранять BOM, если он был в исходном файле, и не добавлять его к файлам без BOM.
10. WHERE применяется нормализация уровня 4 THE SYSTEM SHALL нормализовать конкретный набор: en/em-dash, «умные» кавычки, неразрывный пробел (nbsp), нулевой ширины символы, а также форму Unicode NFC — список фиксируется в дизайне.
11. THE SYSTEM SHALL поддерживать операции патча add-file, delete-file, move/rename; при мультифайловой правке применять её транзакционно (всё или ничего с откатом по снапшоту).
12. WHEN якорь совпал в 0 мест THE SYSTEM SHALL возвращать actionable-ошибку с фрагментом искомого контекста и подсказкой уточнить якорь; файл остаётся байт-в-байт неизменным.
13. IF файл бинарный/не-UTF-8 THE SYSTEM SHALL отказывать в текстовой правке с понятным сообщением, а не портить содержимое.
14. WHEN правка является no-op (old==new или пустой diff) THE SYSTEM SHALL отклонять её как ошибку.
15. THE SYSTEM SHALL выполнять запись файла атомарно (temp-файл + rename), чтобы прерывание не оставляло полу-записанный файл.
16. THE SYSTEM SHALL применять чёткое правило выбора: `write_file` (полная перезапись) при размере файла ≤ порога (по умолчанию ~400 строк) или создании нового; `edit_file` (точечная правка) иначе.

### Requirement 4: Набор инструментов (ACI)

**User Story:** Как агент, я хочу небольшой, LM-дружелюбный набор высокоуровневых инструментов с предсказуемым контрактом, чтобы точно и дёшево исследовать и менять проект.

#### Acceptance Criteria
1. THE SYSTEM SHALL предоставлять инструменты: `list_dir`, `read_file` (диапазоны строк), `grep_search` (ripgrep), `find_path` (glob), `edit_file` (точечная замена), `write_file` (перезапись), `run_command`, и `diagnostics` (если доступен LSP/линтер).
2. THE SYSTEM SHALL определять инструменты через `tool()` + Zod `inputSchema`, чтобы ввод модели валидировался автоматически.
3. THE SYSTEM SHALL возвращать из каждого инструмента единый контракт `{ title (для UI), output (для модели), metadata }`.
4. WHEN вывод инструмента превышает лимит THE SYSTEM SHALL усекать его как head+tail с явным маркером «[вывод обрезан: X→Y]», а не молча.
5. WHEN независимые read-only инструменты можно выполнить параллельно THE SYSTEM SHALL поддерживать пакетный вызов (`batch`) с частичным успехом.
6. WHEN исключение инструмента THE SYSTEM SHALL превращать его в `tool-error`/`tool.complete{error}` (модель может исправиться), а НЕ ронять цикл.
7. THE SYSTEM SHALL поддерживать долгие/фоновые команды (start→id, инкрементальный вывод, kill) с прокидыванием `abortSignal`.
8. WHEN усекается вывод THE SYSTEM SHALL резать по границам строк/codepoint (не рвать многобайтные символы) и указывать точный маркер с исходным и итоговым размером.
9. WHEN инструмент возвращает не-текстовый результат (изображение) THE SYSTEM SHALL отдавать модели корректное мультимодальное представление (`toModelOutput`), а UI — превью.
10. WHEN завершается фоновая команда/убивается процесс THE SYSTEM SHALL корректно убивать всё дерево процессов (на Windows — `taskkill /T /F`, на POSIX — kill process group).
11. THE SYSTEM SHALL реализовывать `grep_search` через ripgrep (bundled бинарь на всех 3 ОС + fallback на системный/JS) и `find_path` с glob-семантикой.

### Requirement 5: Управление контекстом и токен-бюджет

**User Story:** Как пользователь длинных сессий, я хочу, чтобы движок умно управлял контекстом без потери важного, чтобы качество не деградировало и стоимость не взрывалась.

#### Acceptance Criteria
1. THE SYSTEM SHALL оценивать использование контекста по **двойному триггеру**: локальный подсчёт токенов + `usage` от провайдера.
2. WHEN контекст приближается к порогу THE SYSTEM SHALL выполнять двухфазную компакцию: (1) prune старых tool-outputs, (2) LLM-суммаризация с пометкой точки компакции.
3. THE SYSTEM SHALL создавать инкрементальные чекпоинты рано (ориентир ~20/45/70% окна), а не только у предела.
4. WHEN сжатие/eviction применяется THE SYSTEM SHALL делать его **обратимым** (оригинал доступен по идентификатору через инструмент восстановления — паттерн CCR), не теряя данные безвозвратно.
5. THE SYSTEM SHALL сохранять стабильный префикс (system-prompt, определения инструментов, старые ходы не мутируются) для эффективности prompt-cache провайдера.
6. WHEN достигнут истинный предел окна THE SYSTEM SHALL пересобирать контекст из файлов-seed с ограничением на секцию (bounded per-section), а не тащить всю историю.

### Requirement 6: Память (слоёная, локальная, обратимая)

**User Story:** Как пользователь, я хочу, чтобы агент помнил проект между сессиями и переносил знания в «чистое окно», чтобы не начинать каждый раз с нуля.

#### Acceptance Criteria
1. THE SYSTEM SHALL хранить память как markdown-файлы (source of truth, human-readable, git-версионируемые), а SQLite/индекс использовать как ускоритель поиска.
2. THE SYSTEM SHALL поддерживать слои памяти: сессия (checkpoint) → проект (`MEMORY.md`/steering/`AGENTS.md`) → глобальный.
3. WHEN ведётся структурная память THE SYSTEM SHALL делать это отдельным шагом/writer-ролью (не основным циклом), с инвариантом single-writer и enforced-в-коде путями записи.
4. WHEN задача завершает окно THE SYSTEM SHALL формировать distilled handoff-артефакт и стартовать следующий этап в чистом окне (перечитывая артефакт, а не всю историю чата).
5. THE SYSTEM SHALL интегрироваться с уже существующим слоем `ltm/` как стартовой базой памяти.
6. WHEN обновляется долгая память в idle THE SYSTEM SHALL применять только изменения, прошедшие валидацию (held-out gate), с bounded add/delete/replace-правками (защита от деградации).

### Requirement 7: Провайдеры и маршрутизация моделей

**User Story:** Как пользователь, я хочу гибко подключать много провайдеров и локальные модели, с надёжным фолбэком и учётом стоимости, чтобы «поставил и работает», а сбои не ломали ход.

#### Acceptance Criteria
1. THE SYSTEM SHALL вести реестр моделей (лимиты токенов, стоимость, возможности: streaming/tools/vision).
2. THE SYSTEM SHALL поддерживать роли моделей по интенту (например default/small/plan) и выбор провайдера/модели строкой через реестр.
3. WHEN основной провайдер отвечает ошибкой/недоступен THE SYSTEM SHALL применять fallback-цепочку (proactive + reactive) с экспоненциальным backoff.
4. WHEN у аккаунта несколько ключей THE SYSTEM SHALL поддерживать round-robin с session-affinity (ради prompt-cache) и circuit-breaker/cooldown при сбоях.
5. WHEN провайдер не поддерживает function-calling (400/404/422 на tools) THE SYSTEM SHALL деградировать до вызова без инструментов, а не падать.
6. THE SYSTEM SHALL хранить ключи/учётки в OS-keychain или зашифрованно (AES-256-GCM) и НЕ отдавать их в renderer/логи.
7. THE SYSTEM SHALL использовать ТОЛЬКО легальные способы доступа (официальные API/OAuth); browser-account-bridging (обход ToS) НЕ реализуется.
8. THE SYSTEM SHALL оценивать токены пер-провайдер/модель (разные токенизаторы: OpenAI o200k/cl100k, Llama, Claude — где локального токенизатора нет, использовать откалиброванную эвристику с запасом).
9. WHEN используется потоковый режим THE SYSTEM SHALL запрашивать usage у провайдера (`stream_options.include_usage` для OpenAI-совместимых) и корректно обрабатывать случаи полного/частичного/отсутствующего `usage`.

### Requirement 8: Безопасность исполнения

**User Story:** Как пользователь, я хочу, чтобы агент не мог навредить за пределами рабочей папки и всегда спрашивал на опасное, чтобы доверять автономному режиму.

#### Acceptance Criteria
1. THE SYSTEM SHALL применять workspace-jail с резолвом симлинков (`realpath`) для файловых инструментов; выход за пределы папки запрещён. THE SYSTEM SHALL честно документировать, что для `run_command` джейлится только CWD и применяется deny-list+approval, а полное сдерживание содержимого команды невозможно без OS-песочницы (см. дизайн, «Honest limits»).
2. THE SYSTEM SHALL поддерживать уровни разрешений allow/ask/deny и двухосевую автономию (политика терминала × политика ревью).
3. WHEN выполняется разрушительное/необратимое действие THE SYSTEM SHALL по умолчанию требовать подтверждения (никакого YOLO по умолчанию).
4. THE SYSTEM SHALL стремиться ограничивать сеть для команд агента (network default-deny + allowlist), ЧЕСТНО документируя, что в Node без OS-песочницы это не гарантируется, и предлагая опциональный контейнер/OS-sandbox для строгой изоляции.
5. THE SYSTEM SHALL логировать (аудит) каждый вызов инструмента с параметрами и результатом-статусом.
6. WHEN читается потенциально секретный файл (`.env`, ключи) THE SYSTEM SHALL не выводить значения секретов в контекст/логи.
7. THE SYSTEM SHALL поддерживать pre-tool-use gate (хук), способный заблокировать вызов или просканировать секреты до исполнения.
8. THE SYSTEM SHALL трактовать содержимое читаемых файлов, вывод команд и память как НЕДОВЕРЕННЫЕ данные (защита от prompt-injection): инструкции из них не исполняются как команды агента без явного подтверждения.
9. THE SYSTEM SHALL не пробрасывать секреты окружения в `run_command` по умолчанию (минимизированный/очищенный env) и не сохранять значения секретов в append-only JSONL/аудит.
10. THE SYSTEM SHALL редактировать значения секретов во ВСЕХ каналах вывода: чтение файлов, stdout/stderr команд, `inline_diff`, аудит-лог, события и персист (не только `.env`).
11. THE SYSTEM SHALL требовать подтверждения перед сетевыми/деструктивными действиями и валидировать `baseURL` провайдера (защита от SSRF на внутренние адреса, где применимо).

### Requirement 9: Надёжность агентного цикла

**User Story:** Как пользователь, я хочу, чтобы цикл не зависал, не завершался «оптимистично» и переживал ошибки/отмену без порчи истории.

#### Acceptance Criteria
1. THE SYSTEM SHALL иметь явные условия остановки: финальный ответ, `max-steps`, таймаут, неустранимая ошибка, детект зацикливания.
2. WHEN ход отменён или упал THE SYSTEM SHALL очищать «висячие» сообщения (assistant с tool_calls без tool-результатов), чтобы история осталась валидной для следующего запроса.
3. WHEN включён автономный режим THE SYSTEM SHALL проверять условие завершения независимой верификацией (Goal-verifier) перед «done».
4. WHEN задача проваливается THE SYSTEM SHALL применять протокол self-heal (probe→retry → fix → handoff человеку) с лимитом попыток.
5. WHEN многошаговая задача заявлена выполненной THE SYSTEM SHALL проверять результат (proof/evidence-gated: сборка/тесты/линт), а не доверять заявлению модели.
6. THE SYSTEM SHALL обеспечивать лимиты стоимости/ресурсов: бюджет токенов/стоимости на ход, лимит числа/глубины суб-агентов, таймаут команды, детект runaway (кратный повтор без прогресса) с остановкой.
7. WHEN детектируется зацикливание THE SYSTEM SHALL опираться на условие «нет прогресса» (повтор тех же tool-call/результатов), а не только на счётчик, чтобы избегать ложных срабатываний.

---

### Requirement 10: Слой данных

**User Story:** Как пользователь local-first, я хочу, чтобы всё хранилось локально в переносимом виде, с опцией масштабирования, чтобы данные принадлежали мне.

#### Acceptance Criteria
1. THE SYSTEM SHALL использовать по умолчанию SQLite (WAL) + sqlite-vec (вектора) + FTS5 (keyword), всё embedded и офлайн.
2. THE SYSTEM SHALL хранить транскрипт сессии как append-only JSONL (source of truth), а SQLite — как индекс/срез поверх.
3. THE SYSTEM SHALL прятать доступ к данным за port-интерфейсами (`SessionStore`/`MemoryStore`/`VectorStore`), чтобы бэкенд менялся конфигом.
4. WHERE пользователь выбирает PostgreSQL THE SYSTEM SHALL поддерживать его как опциональный бэкенд (+ pgvector) за тем же интерфейсом, без изменения доменной логики.
5. WHEN тяжёлая операция БД (bulk-embedding, миграция) THE SYSTEM SHALL выполнять её вне главного event-loop (worker), чтобы не блокировать UI.

### Requirement 11: Оркестрация и суб-агенты

**User Story:** Как пользователь сложных задач, я хочу, чтобы движок умел планировать и параллелить чтение/исследование, но безопасно писать одним агентом.

#### Acceptance Criteria
1. THE SYSTEM SHALL исполнять записи/правки строго одним агентом-писателем (никаких параллельных писателей).
2. WHERE задача read-ориентированная (поиск/исследование) THE SYSTEM SHALL позволять read-рой суб-агентов с изолированным контекстом, возвращающих компактные саммари.
3. THE SYSTEM SHALL поддерживать ревьюера с чистым контекстом, который видит только дифф.
4. WHEN сложная задача декомпозируется THE SYSTEM SHALL хранить план как файлы (`ROADMAP/STATE/phase-N`) с адаптивным числом фаз и end-state-условием.
5. WHEN суб-агент возвращает результат THE SYSTEM SHALL валидировать его по схеме; суб-агенту-исполнителю запрещено ре-делегировать (защита от бесконечной делегации).

### Requirement 12: Тестирование и «доведение до идеала»

**User Story:** Как мейнтейнер, я хочу исчерпывающий тест-набор и eval-харнесс, чтобы объективно доводить движок до идеала и не ловить регрессии.

#### Acceptance Criteria
1. THE SYSTEM SHALL иметь unit-тесты на все чистые функции (apply/seek, diff, token-estimator, safePath/jail, compaction, provider-registry, cleanup-incomplete).
2. THE SYSTEM SHALL иметь integration-тесты цикла на штатных тест-утилитах AI SDK (`MockLanguageModelV2` + `simulateReadableStream` из `ai`/`ai/test`) — без сети и без локального сервера: стрим text/reasoning-дельт, multi-step tools (≥2 хода), стабильный `tool_call_id`, отмена→interrupted, usage в `message.complete`.
3. THE SYSTEM SHALL иметь property-based тесты на apply-движок (случайные правки → применение → эквивалентность/идемпотентность) и на jail (никакой путь не выходит за папку).
4. THE SYSTEM SHALL иметь provider-matrix прогон (OpenAI/DeepSeek/OpenRouter/локальный Ollama или mock) для стрима/tools/cancel/usage.
5. THE SYSTEM SHALL иметь eval-харнесс на наборе кодинг-задач с метриками (успех правки, число шагов, токены) для сравнения версий движка.
6. WHEN изменяется движок THE SYSTEM SHALL прогонять сборку + `node --check`/tsc + тесты и оставаться «зелёным» до мержа.
7. THE SYSTEM SHALL держать безопасность-тесты: jail-escape (симлинки/`..`/абсолютные пути/Windows UNC/`\\?\`/junctions), network-deny, секрет-редакция, approval-gate на разрушительное, prompt-injection из недоверенного контента.
8. THE SYSTEM SHALL иметь отдельный провайдер-контрактный тест реального пути `createOpenAICompatible({ baseURL, fetch })` через инъекцию кастомного `fetch` (recorded fixtures): парсинг SSE wire-формата, парсинг usage (полный/частичный/отсутствует), no-tools-fallback на HTTP 400/404/422.
9. THE SYSTEM SHALL прогонять весь тест-набор под обоими значениями `KYREI_ENGINE` (v1|v2) и на CI-matrix (Windows/macOS/Linux) вплоть до Фазы 7; property-тесты — с фиксированным `seed` и `numRuns`.

### Requirement 13: Доказуемое превосходство (measurable acceptance)

**User Story:** Как владелец продукта, я хочу измеримые критерии «лучше аналогов», чтобы превосходство движка было доказуемым, а не декларативным.

#### Acceptance Criteria
1. THE SYSTEM SHALL достигать `edit_success` ≥ 95% на eval task-set (E1–E6) в детерминированном CI-режиме.
2. THE SYSTEM SHALL показывать медиану `steps` и `tokens` на идентичном task-set не хуже v1 (регресс > 20% = fail).
3. THE SYSTEM SHALL держать 0 «молчаливых» неверных правок (Property уникальности якоря/обратимости выдерживают ≥ 1000 property-прогонов).
4. THE SYSTEM SHALL держать 0 jail-escape и 0 secret-leak на security-наборе.
5. THE SYSTEM SHALL проходить verification-gate на всех трёх ОС.
6. THE SYSTEM SHALL фиксировать превосходство артефактом `tests/eval/report.md` (v1 vs v2 + дельты), обновляемым в каждом релизе.

## Нефункциональные требования / границы

- **Кроссплатформенность:** Windows/macOS/Linux; shell-команды и пути учитывают ОС.
- **Производительность:** ядро цикла I/O-bound (не оптимизируем преждевременно); CPU-hot подсистемы (индекс/поиск/дифф) — кандидаты на Rust/napi-rs позже (§29).
- **Совместимость:** renderer и формат событий не ломаются (мост `fullStream → emit`).
- **Вне рамок (пока):** полноценная OS-песочница на 3 ОС «с нуля»; MCP-клиент (фаза позже); rich-composer; полный spec-режим UI.
