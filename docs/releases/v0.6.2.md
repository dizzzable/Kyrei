# Kyrei v0.6.2 — uninterrupted agent recovery

This patch prevents Kyrei's internal reliability guardrails from ending a
user's task while useful recovery paths still exist.

## Fixed

| | |
| --- | --- |
| **No false three-strike handoffs** | Parallel tool failures from one model step are counted as one failed attempt, not several consecutive retries. Any successful tool result in the batch restores forward progress. |
| **Automatic clean recovery** | Step limits, repeated tool calls, heal handoffs, and an unsatisfied goal now create a private checkpoint and continue the same logical turn with a clean recovery pass. They no longer publish a premature completion event or force the user to restart the task. |
| **Context survives recovery** | The next pass keeps the conversation, tool results, accumulated response parts, provider attempts, and cumulative token and cost accounting. Explicit user-configured budgets remain respected. |
| **Private diagnostics stay private** | Internal `KYREI_FAILURE_HANDOFF` markers and local handoff paths are filtered from streamed output and durable chat storage, including markers split across stream chunks. Existing affected sessions are sanitized when loaded. |

## Behaviour after upgrading

- Kyrei continues automatically when an internal guardrail requests a clean
  recovery pass.
- There is no hidden recovery-pass cap. A task still stops for explicit user
  cancellation, approval or file-review boundaries, an external provider
  failure that cannot be recovered, or a token/cost budget configured by the
  user.
- Updates are in-place: sessions, provider secrets, project memory, indexes,
  and user settings remain untouched.
