# Kyrei v0.5.1 — Stable local health, MCP handshakes, complete updates

This patch release removes the repeated memory-health probes that made the
index, mirror, and GBrain cards oscillate between available and unavailable.
It also makes MCP diagnostics actionable and ships complete dependency bundles
through the updater.

## Highlights

| | |
| --- | --- |
| **Stable memory health** | Settings health callbacks are identity-stable; gateway probes share in-flight work and cache short-lived results while reindexing. Session-mirror initialization is race-safe. |
| **MCP that can connect** | Stdio accepts current newline-delimited MCP JSON-RPC as well as legacy `Content-Length` frames, resolves Windows npm-family `.cmd` launchers, captures bounded stderr diagnostics, and exposes per-server tool counts in Settings. |
| **No more raw `Failed to fetch`** | Local gateway network failures map to a stable user-facing state so LTM decisions can be retried without exposing browser fetch internals. |
| **Complete application updates** | CI uses Node 24, publishes updater metadata/blockmaps, and the installed updater forces a full bundle replacement so the renderer, engine, native modules, and dependency graph move together. |
| **Update visibility and icon polish** | Startup checks run once, the bottom-right status indicator glints when an update is available, and all NSIS icon surfaces point at the generated Kyrei icon. |

## Upgrade notes

- Safe in-place upgrade from **v0.5.0**; sessions, provider secrets, and memory files remain on disk.
- Existing MCP configurations remain valid. Open Settings → Memory → MCP servers and press **Check** to see each process and its discovered tool count.
- Configure at least one enabled server; enabling MCP alone intentionally reports “no servers”.
- The updater downloads a complete installer rather than applying a differential patch. This is slower but avoids stale native dependency graphs after an update.
